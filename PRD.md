# 开源记账工具 PRD

## 1. 产品概述

### 1.1 产品名称（暂定）
PocketLedger

### 1.2 产品定位
一个面向大学生、本地优先、开源、可从 GitHub 下载并安装的桌面端个人账本工具。

### 1.3 一句话描述
帮助大学生围绕月生活费管理日常收支、账户余额和消费结构，清楚知道钱花到了哪里、还剩多少。

## 2. 背景与目标

### 2.1 背景
现有很多记账产品存在以下问题：

- 过度依赖云端和账号体系，用户对隐私和数据所有权不放心。
- 功能过重，包含预算、社区、理财、广告、会员等复杂模块，导致核心记账体验变差。
- 很多产品更偏向成年上班族或家庭财务，不够贴近大学生的真实消费场景。
- 很多产品更像消费内容平台，不像一个安静、可信的个人账本。
- 安装和迁移不方便，不适合开源方式长期维护。

### 2.2 产品目标

- 做一个真正可用的大学生个人账本 MVP，而不是演示页面。
- 默认本地存储，用户不必注册登录。
- 围绕“月生活费”建立核心体验，而不是只做流水记录。
- 让用户从账户维度理解自己的消费习惯，例如微信、支付宝、银行卡分别花了多少。
- 开源托管在 GitHub，源码可 clone，Release 可下载安装包。
- 后续可逐步扩展，但第一版先把“记清楚校园生活里的账”这件事做好。

### 2.3 非目标

- 第一版不做多人协作。
- 第一版不做银行自动同步。
- 第一版不做复杂预算系统。
- 第一版不做投资组合、报税、发票识别等重模块。
- 第一版不做 SaaS 云同步。
- 第一版不做移动端 App。

## 3. 目标用户

### 3.1 核心用户

- 有固定生活费或有限月度预算的大学生。
- 想知道“这个月还剩多少钱能花”的学生。
- 希望搞清楚自己钱花到哪了，但不想用复杂理财产品的学生。
- 喜欢开源工具，希望能自己备份、导出、二次开发的学生用户。

### 3.2 核心需求

- 记账足够快，最好能在几秒内记下一笔奶茶、外卖、交通或购物支出。
- 能清楚看到本月生活费已经花了多少，还剩多少。
- 能区分钱是从微信、支付宝、银行卡还是现金花出去的。
- 能按分类查看消费分布，知道钱主要花在吃饭、出行还是娱乐上。
- 数据能导出，能备份，不被平台锁死。

### 3.3 典型使用场景

- 食堂、外卖、奶茶、快递、打车、超市购物等高频小额支出记录。
- 每月复盘生活费使用情况，避免“钱不知道花哪了”。
- 兼职、奖学金、家里转账等收入记录。
- 微信、支付宝、银行卡之间转账后同步更新账本。
- 学期结束时导出数据，做个人消费总结。

## 4. 核心价值主张

- 本地优先：数据默认保存在本机。
- 学生场景优先：围绕生活费、校园消费和兼职收入设计，而不是照搬成人理财产品。
- 轻量直接：避免复杂流程，专注大学生日常收支记录。
- 开源透明：代码和数据结构公开，便于信任和维护。
- 可安装：用户可以从 GitHub Release 下载安装，而不是只会跑源码。

## 5. MVP 范围

### 5.1 必做功能

#### A. 交易管理

- 新增一条交易。
- 编辑一条交易。
- 删除一条交易。
- 交易类型包括：
  - 收入
  - 支出
  - 转账
- 交易字段包括：
  - 类型
  - 金额
  - 分类
  - 账户
  - 日期
  - 备注

#### B. 账户系统

- 提供默认账户。
- 支持用户新增、编辑、删除账户。
- 支持设置账户初始余额。
- 支持账户之间转账。
- 支持查看各账户当前余额。

建议默认账户：

- 微信
- 支付宝
- 银行卡
- 现金

#### C. 分类系统

- 提供默认分类。
- 支持用户新增自定义分类。
- 区分收入分类和支出分类。

建议默认分类：

- 支出：食堂、外卖、奶茶饮料、交通、购物、宿舍生活、娱乐社交、学习资料、医疗、其他
- 收入：生活费、兼职、奖学金、退款、红包、其他

#### D. 首页总览

- 快速记一笔入口（首页最高优先级）
- 本月生活费
- 本月生活费已用
- 本月生活费剩余
- 当前月总支出
- 当前月总收入
- 当前月结余
- 账户余额概览
- 最近交易列表
- 快速记账入口

#### E. 统计分析

- 按月查看收支汇总
- 按分类查看支出占比
- 按账户查看支出分布
- 按日期查看近 30 天支出趋势

#### F. 筛选与搜索

- 按月份筛选
- 按类型筛选
- 按分类筛选
- 按账户筛选
- 按备注关键词搜索

#### G. 数据导入导出

- 导出为 JSON
- 导出为 CSV
- 从 JSON 导入

#### H. 本地持久化

- 数据默认保存在本地文件
- 自动保存
- 提供手动备份入口

### 5.2 可选但重要的增强项

- 深色模式
- 快捷键快速记账
- 标签系统
- 超支提醒
- 多账本

### 5.3 暂不进入 MVP

- 云同步
- 移动端 App
- OCR 小票识别
- 自动记账规则
- 多币种
- 家庭共同记账
- 银行卡自动同步

## 6. 关键使用流程

### 6.1 首次使用

1. 用户从 GitHub Release 下载并安装应用。
2. 打开应用后进入一个不超过 3 步的简短引导页。
3. 系统预置默认分类和默认账户。
4. 用户设置本月生活费金额。
5. 用户补充自己的常用账户余额，或直接跳过。
6. 用户点击“新增交易”，录入第一笔支出或收入。

### 6.2 日常记账

1. 用户打开应用。
2. 首页默认突出显示“记一笔”入口。
3. 点击后从右侧滑出快速记账抽屉，不打断当前页面上下文。
4. 输入金额、类型、分类、账户、日期、备注。
5. 保存后立即更新生活费进度、账户余额和统计信息。
6. 若超支则在抽屉顶部和首页摘要区给出明显但非阻断式提醒。

### 6.3 月度复盘

1. 用户切换到某个月份。
2. 查看该月生活费使用情况、收入、支出、结余。
3. 查看分类占比、账户分布、转账总额和趋势图。
4. 找出主要花销类别和最常使用的支付账户。

### 6.4 转账与账户管理

1. 用户在设置中新增或编辑账户。
2. 用户发起账户间转账，例如银行卡转支付宝。
3. 系统自动更新两个账户余额，并保留转账记录。
4. 若用户手动校准余额，系统额外记录一条校准日志以便追溯。

### 6.5 数据备份与迁移

1. 用户点击“导出数据”。
2. 生成 JSON 或 CSV 文件。
3. 在新设备上通过导入恢复账本。

### 8.5 关键规则

- 月生活费按自然月统计，不支持自定义周期作为首版核心能力。
- 账户余额允许用户手动校准，以处理漏记、退款或账本与真实余额不一致的情况。
- 转账记录默认展示在账本中，并在统计页单独作为“转账”模块展示，避免被误读为收入或支出。
- 首版提供明显但非阻断式超支预警，当本月支出达到生活费的 80% 时给出预警，超过生活费时给出更强提示。
- 快速记一笔采用右侧抽屉交互，以保持首页上下文完整并兼顾桌面端操作效率。
- 首页默认展示最近 6 条交易，既能快速回看，又不压缩生活费和账户模块的可视空间。
- 账户手动校准需要保留一条可追溯的校准记录，包含调整前余额、调整后余额、时间和备注。
- 快速记账抽屉默认展开金额、类型、分类、账户和日期字段，备注折叠在“更多选项”中。

## 9. 平台与技术假设

### 9.1 形态选择
桌面应用优先。

原因：

- 更符合“下载安装”的预期。
- 更适合本地文件存储。
- 更容易通过 GitHub Release 直接分发安装包。
- 更适合做成一个安静、长期使用的个人账本工具。

### 9.2 技术方案（建议）

- 前端：React + Vite
- 桌面壳：Electron
- 本地存储：JSON 文件，后续可升级为 SQLite
- 打包发布：electron-builder
- CI/CD：GitHub Actions 自动构建 Release

### 9.3 为什么第一版不用 SQLite

- JSON 足够支撑 MVP。
- 开发快，调试简单，备份直观。
- 大学生个人账本数据量通常不大，首版没有必要引入更重的数据层。
- 后续若账目量变大，再迁移到 SQLite。

### 10.3 开源策略建议

- 优先使用 MIT License，降低使用门槛
- 首版强调个人使用、可 fork、欢迎贡献

## 11. 体验原则

- 记账动作必须快，最好 10 秒内完成一笔录入。
- 生活费信息必须醒目，用户打开应用就知道这个月还能花多少。
- 首页第一优先级是“快速记一笔”，而不是复杂分析。
- 账户信息要可信，用户能明确看到微信、支付宝、银行卡和现金余额变化。
- 视觉上要克制、清楚，不做花哨的理财平台感，更像一个干净的学生效率工具。
- 交互上优先用抽屉和卡片，不频繁跳页或弹出阻断式对话框。
- 警告信息要明确，但不能制造焦虑感或打断记账动作。
- 统计结果要一眼能看懂，而不是堆复杂图表。
- 空状态要友好，让第一次打开时知道下一步做什么。

## 12. 界面风格建议

### 12.1 视觉方向

- 整体风格采用“安静的校园效率工具”，而不是金融平台或社交产品。
- 以浅色为默认主题，保留纸张感和轻微层次，不使用夸张霓虹或理财大屏风格。
- 主色建议偏墨绿或深青，传达克制、清楚、可信的气质。
- 超支和风险提示使用暖红或琥珀色，只在必要位置出现。

### 12.2 组件语言

- 首页用大卡片承载生活费、收支和账户摘要，形成强信息层级。
- 快速记账用右侧抽屉，输入区保持紧凑，常用字段优先展示。
- 账户列表用简洁余额卡展示，不做拟物钱包效果。
- 图表数量控制在 2 到 3 个核心图，不堆砌可视化。
- 图表默认采用极简静态风格，只保留轻微入场动画和悬停反馈。

### 12.3 信息优先级

首页建议信息顺序：

1. 快速记一笔
2. 本月生活费与剩余金额
3. 本月支出 / 收入 / 结余
4. 账户余额概览
5. 最近 6 条交易
6. 简要趋势或分类摘要

### 12.4 新手引导

- 首版提供一个简短引导页，只覆盖生活费设置、账户初始化和第一笔交易提示。
- 引导必须可跳过，避免阻碍已经熟悉记账流程的用户。
- 引导完成后不再频繁弹出，仅在清空数据或首次安装时出现。

## 17. 我建议的结论

如果你的目标是尽快做出一个真正能发到 GitHub 的开源工具，那么当前最合理的路线不是先追求大而全，而是：

- 先做桌面端
- 先做本地优先
- 先做单用户
- 先做月生活费、账户、收支、分类、统计、导入导出
- 先把安装包和 GitHub Release 跑通

这条路线最稳，也最容易在短时间内做出一个像样的开源产品。
